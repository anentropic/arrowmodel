Add a second benchmark function `run_nested_benchmark(sizes)` to benchmarks/bench_convert.py that measures arrowdantic vs to_pylist+model_construct on deeply nested complex data. Keep the existing `run_benchmark` function unchanged.

Define Pydantic models for the nested schema. The goal is 10 levels of nesting with a mix of types:

```python
class Level10(BaseModel):
    value: int
    label: str

class Level9(BaseModel):
    child: Level10 | None
    score: float

class Level8(BaseModel):
    child: Level9 | None
    tags: list[str] | None = None

class Level7(BaseModel):
    child: Level8 | None
    count: int

class Level6(BaseModel):
    child: Level7 | None
    active: bool

class Level5(BaseModel):
    child: Level6 | None
    name: str

class Level4(BaseModel):
    child: Level5 | None
    ids: list[int] | None = None

class Level3(BaseModel):
    child: Level4 | None
    ratio: float

class Level2(BaseModel):
    child: Level3 | None
    code: str

class Level1(BaseModel):
    child: Level2 | None
    timestamp: float

class NestedBenchModel(BaseModel):
    id: int
    name: str
    data: Level1 | None
```

Create `make_nested_batch(n: int) -> pa.RecordBatch` that builds this schema using pyarrow. Build the struct arrays bottom-up:
- Level10: struct with int32 "value" and utf8 "label"
- Level9: struct with Level10 struct "child" and float64 "score"
- Level8: struct with Level9 struct "child" and list(utf8) "tags" (3 items per row)
- Level7: struct with Level8 struct "child" and int32 "count"
- Level6: struct with Level7 struct "child" and bool "active"
- Level5: struct with Level6 struct "child" and utf8 "name"
- Level4: struct with Level5 struct "child" and list(int32) "ids" (5 items per row)
- Level3: struct with Level4 struct "child" and float64 "ratio"
- Level2: struct with Level3 struct "child" and utf8 "code"
- Level1: struct with Level2 struct "child" and float64 "timestamp"
- Top level: int64 "id", utf8 "name", Level1 struct "data"

Use pa.StructArray.from_arrays to build each level. Every field should be non-null for simplicity (no null masks) so the benchmark focuses on nesting depth, not null handling.

For the data generation pattern: use deterministic data based on row index i:
- Strings: f"val_{i}", f"tag_{i}_{j}", f"name_{i}", f"code_{i}"
- Ints: i, i*10, etc.
- Floats: float(i) * 0.1, float(i) * 1.5
- Bools: i % 2 == 0
- Lists: [f"t_{i}_0", f"t_{i}_1", f"t_{i}_2"] for tags, list(range(i, i+5)) for ids

The `run_nested_benchmark(sizes)` function should follow the exact same timing pattern as `run_benchmark`:
- Same ROUNDS calculation: `rounds = max(10, ROUNDS // max(1, n // 1000))`
- Warmup both paths first
- Time arrowdantic: `converter.convert(batch)` with `ArrowModelConverter(NestedBenchModel)`
- Time baseline: `[NestedBenchModel.model_construct(**row) for row in batch.to_pylist()]`
  NOTE: The baseline model_construct for nested data only constructs the top-level model -- it does NOT recursively construct nested models from dicts. This is a known limitation of the baseline. The benchmark is still valid because it shows the cost difference of the entire operation, and in practice users need the nested models too.
- Print the same formatted table with header "Nested (10-level struct + lists)"

Update the `if __name__ == "__main__"` block to run both benchmarks:
```python
if __name__ == "__main__":
    sizes = [int(arg) for arg in sys.argv[1:]] if len(sys.argv) > 1 else DEFAULT_SIZES

    run_benchmark(sizes)
    print("\n=== Nested (10-level struct + lists) ===\n")
    run_nested_benchmark(sizes)
```

Use smaller DEFAULT_SIZES for the module: keep [1000, 10_000, 100_000] but for the nested benchmark, consider that deeply nested data is much slower to create and convert. The same sizes list is passed -- the rounds adjustment formula already handles this.
